# DATA_MODEL.md

## Overview

PULO uses PostgreSQL via Drizzle ORM. The schema is designed to support a multi-tenant SaaS product with a growing set of AI-powered features, without requiring rewrites as new capabilities are added.

## Design Principles

- **Normalization first** — Avoid JSON columns for fields that are frequently queried. Use JSONB only for flexible metadata, raw payloads, and rarely-filtered fields.
- **Timestamps everywhere** — All tables have `created_at`. Tables with mutable records have `updated_at`.
- **Soft deletes preferred** — Status columns (user_status, trend_status, etc.) over hard deletes for auditability.
- **UUID for events, SERIAL for users** — Events/audits/runs are UUID so distributed systems can generate IDs without coordination. Users use SERIAL for compactness and simple FK relationships.
- **Idempotency keys everywhere** — Alert deliveries, events, and rate limit records all carry idempotency/dedupe keys to prevent double-processing.
- **Indexes on query paths** — Every foreign key and filter field has an index.

## Table Inventory

### users
Central identity table. `fid` is the Farcaster FID and is the primary business key. One row per user, created on first interaction.

### user_preferences
Per-user settings for tone, reply style, notification frequency, alert thresholds, and automation preferences. One row per user (upsert pattern).

### subscriptions
Tracks subscription tier and provider. Supports Stripe, Paddle, LemonSqueezy, or manual overrides. `expires_at` determines active status.

### casts
Normalized cast records. Deduplication via `cast_hash` UNIQUE constraint. `raw_json` stores the full Neynar payload for replay/debug.

### cast_threads
Thread-level aggregation. `root_hash` references the top-level cast. `cast_hashes` array enables thread reconstruction without recursive queries.

### agent_events
**Event sourcing table** — all inbound interactions (mentions, DMs, scheduled scans) create an event here. `dedupe_key` enables idempotent processing. `status` drives the worker queue.

### agent_runs
LLM invocation records. Each run captures model, tokens, cost, decision, and output. Enables usage analytics, cost tracking, and quality evaluation.

### reply_drafts
Draft replies generated by the LLM before user approval or auto-publish. `score` is the LLM quality score (0-100). `published_cast_hash` links to the actual posted cast.

### truth_checks
Fact-check records. Stores the claim, verdict, confidence (0-100), evidence summaries, and source cast hashes. Powers the "!pulo is..." analysis feature.

### trends
Aggregated trend tracking. `score` is a composite of velocity × confidence × source diversity. `status` drives radar filtering (active vs fading).

### trend_sources
Source-level evidence for trends. Each row links a cast to a trend with engagement and trust scoring. Enables source-level drill-down.

### alert_deliveries
Delivery log with idempotency key. Tracks delivery status (sent/delivered/opened/failed) and timing. Powers notification analytics.

### admin_audit_logs
Immutable audit trail for admin actions. `before`/`after` JSON columns capture delta. `actor_user_id` may be null for system-initiated actions.

### rate_limit_events
Rate limit decision log. Every allow/deny/throttle decision is recorded. Enables abuse pattern detection and per-user limit tuning.

### safety_flags
Content moderation flags. Links to users, casts, or channels. `status` drives review workflow (active → reviewed → cleared/escalated).

## Index Strategy

| Table | Index | Purpose |
|---|---|---|
| users | `users_fid_idx` (unique) | Lookup by FID |
| users | `users_username_idx` | Username search |
| casts | `casts_cast_hash_idx` (unique) | Cast dedup |
| casts | `casts_author_fid_idx` | User cast history |
| casts | `casts_parent_hash_idx` | Thread building |
| agent_events | `agent_events_dedupe_key_idx` | Idempotency |
| agent_events | `agent_events_status_idx` | Worker queue query |
| agent_events | `agent_events_type_idx` | Event type filtering |
| agent_runs | `agent_runs_user_id_idx` | User usage history |
| agent_runs | `agent_runs_status_idx` | Run status filtering |
| trends | `trends_category_idx` | Category filtering |
| trends | `trends_velocity_idx` | Velocity ranking |
| alert_deliveries | `alert_deliveries_idempotency_key_idx` (unique) | Idempotent delivery |
| admin_audit_logs | `admin_audit_logs_entity_type_idx` | Entity audit search |

## Migrations

Migrations are SQL files in `packages/db/migrations/`. They are applied manually in development:

```bash
PGPASSWORD=password psql -h localhost -p 5544 -U postgres -d pulo -f packages/db/migrations/0000_init_pulo_schema.sql
```

In production, use `pnpm --filter @pulo/db db:migrate` after configuring `DATABASE_URL`.

## Extension Requirements

The following PostgreSQL extensions are used:
- `pgcrypto` (built-in, for `gen_random_uuid()`)
- No external extensions required
